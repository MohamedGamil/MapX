## Objective

Extend mapx from 3 supported languages to 22 by adding tree-sitter-based support for Python, Go, Rust, Java, C#, Ruby, C, C++, Swift, Kotlin, Scala, Dart, Svelte, Vue, Lua/Luau, Elixir, Zig, Bash/Shell, and Pascal/Delphi.

After I12, mapx can meaningfully index virtually any polyglot codebase. An LLM working on a Python + TypeScript project, a Go microservice, a Rust CLI tool, or a Ruby on Rails app all get full symbol graphs, call edges, and import dependencies — not just PHP/JS/TS.

## Implementation phases

Implement in order: refactor first, then build upward from most-used languages.

### Phase 1: GenericWasmParser + refactor existing parsers (prerequisite)

1. Create `src/parsers/generic-wasm-parser.ts` — `GenericWasmParser` base class with default scope tracking and signature extraction
2. Refactor `PhpParser`, `JavaScriptParser`, `TypeScriptParser` to extend `GenericWasmParser` (behavior unchanged, boilerplate removed)
3. Run existing tests to verify no regression
4. Add `LanguageTier` type and `grammarNpm` field to `LanguageDefinition` in `src/languages/registry.ts`
5. Update `LanguageTier` from `'built-in' | 'installable' | 'user'` to `'built-in' | 'bundled' | 'installable' | 'user'`
6. Add grammar resolution order to `loadLanguage()` in `wasm-parser.ts` (project-local → user → bundled)

### Phase 2: Tier 1 languages (Python, Go, Rust, Java, C#)

For each language: write query files → add registry entry → register parser → add to build-wasm.ts

Order within this phase (most to least impactful by GitHub usage):
1. Python
2. Go
3. Java
4. C#
5. Rust

Deliverable: `mapx scan` on sample projects in all 5 languages produces correct symbol graphs.

### Phase 3: Tier 2 languages (Ruby, C, C++, Swift, Kotlin, Scala, Dart)

Same process per language. Most use `GenericWasmParser` directly or thin subclasses.

Order:
1. C (simplest — functions only, no scope tracking)
2. Ruby
3. Kotlin
4. Swift
5. C++ (shares C grammar patterns)
6. Dart
7. Scala

### Phase 4: `mapx lang` CLI commands

Implement `mapx lang install / uninstall / list / info` commands.

- `mapx lang list` works even before any installable languages are downloaded
- `mapx lang install <name>` downloads from CDN or npm registry to `~/.mapx/grammars/`
- Grammar resolution in `loadLanguage()` checks user grammar dir before bundled

### Phase 5: Tier 3 languages (Svelte, Vue, Lua, Elixir, Zig, Bash, Pascal)

These are all installable — WASM not bundled. Implement per-language in this order:
1. Bash (simplest — functions only)
2. Zig (clean grammar)
3. Lua/Luau
4. Elixir
5. Svelte (specialized script-block delegation)
6. Vue (Options API + Composition API)
7. Pascal/Delphi

### Phase 6: Shebang detection + documentation

- Shebang detection in `src/core/scanner.ts` for extensionless shell files
- Update `docs/adding-languages.md` with contributor guide
- Update `docs/cli-reference.md` with `mapx lang` commands
- Update `docs/getting-started.md` with language support table

## Notes

- The biggest engineering risk is **scope tracking** for languages where method scope is implicit or requires traversing multiple AST levels (Go receivers, Rust impl blocks, C++ nested classes). Build test cases for each before claiming complete.
- Svelte and Vue parsers are the most complex due to multi-part file structure. Consider implementing them last in Phase 5 after simpler languages validate the `GenericWasmParser` approach.
- `tree-sitter-pascal` grammar is less maintained than others. If it produces poor results, use the fallback parser for Pascal and document the limitation.
- `tree-sitter-luau` may need to be compiled from source; ensure the build script handles this gracefully.
- `tree-sitter-swift` is a third-party grammar with known parsing gaps for newer Swift features (async/await, macros). Index quality will be "good" not "perfect".
- Fish shell (`.fish`) has a distinct syntax from bash/zsh. Use `tree-sitter-bash` as a best-effort for now; a future iteration can add `tree-sitter-fish` if demand warrants it.
