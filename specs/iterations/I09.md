## Objective

When a developer runs `mapx init` in their project, mapx should generate **provider-specific LLM integration files** that teach their LLM tools how to use mapx for that codebase. These files cover all major LLM and agentic coding tools — Claude, Cursor, GitHub Copilot, Windsurf, Cline, Aider, Gemini CLI, Continue, and Zed — each with correct syntax, placement, and content (including MCP server configuration where supported).

After I09, a developer who installs mapx and runs `mapx init` will have zero-configuration LLM integration: their chosen AI tools immediately understand how to use mapx to explore the codebase.

## Key design decisions

### Append-safe writes for shared files

Some providers use files that may contain other project instructions (`.github/copilot-instructions.md`, `.clinerules`). The generator never overwrites these files wholesale — it appends the `<!-- mapx -->` block and uses the sentinel to manage its own section on future updates.

### Sentinel-based update model

The `<!-- mapx VERSION TIMESTAMP -->` sentinel makes the managed block clearly delimited and machine-parseable. `mapx agents update` diffs only the block content, not the whole file. This allows users to freely edit content outside the block without conflict.

### Templates bundled at build time

Templates live in `src/agents/templates/` and are embedded into the binary at build time. There is no network fetch at runtime. This means:
- Works offline
- Reproducible output
- Template updates ship with mapx version bumps

### `mapx init` integration is non-blocking

The provider selection step during `mapx init` is purely optional and skippable with Enter or `--no-agents`. It does not block or slow down the init process if declined.

### Binary path resolution

`{{MAPX_BIN}}` is resolved via `process.execPath` (the current binary) with a fallback to `which mapx`. This ensures generated MCP configs point to the correct binary, even when installed in non-standard locations.

## Acceptance Criteria (I09 complete when all pass)

### Generation

- [ ] `mapx agents generate --providers=all` creates all 10 provider files
- [ ] Each file contains the `<!-- mapx VERSION -->` sentinel
- [ ] All `{{TEMPLATE_VARS}}` are substituted — no raw placeholders in output
- [ ] `{{MAPX_BIN}}` resolves to actual binary path
- [ ] `--dry-run` writes nothing (0 files changed)
- [ ] `--force` overwrites without prompting
- [ ] Parent directories are created if they don't exist (`.cursor/rules/`, `.windsurf/rules/`, etc.)
- [ ] Append-safe providers (Copilot, Cline) do not overwrite existing content

### Provider correctness

- [ ] `CLAUDE.md` contains both stdio and SSE MCP config stanzas
- [ ] `.cursor/rules/mapx.mdc` has valid YAML frontmatter with `alwaysApply`, `globs`, `description`
- [ ] `.windsurf/rules/mapx.md` has valid frontmatter with `trigger: model_decided`
- [ ] `.continue/mapx.yaml` is valid YAML with `contextProviders` and `slashCommands`
- [ ] `.zed/settings.json` is updated (merged, not replaced) with `assistant.default_context`
- [ ] `AIDER.md` includes the `--read` flag workflow note and `.aider.conf.yml` snippet

### Listing

- [ ] `mapx agents list` detects all generated files
- [ ] `mapx agents list` reports `✓ current` for version-matched files
- [ ] `mapx agents list` reports `✗ outdated` for older-version files
- [ ] `mapx agents list` reports `⚠ manually edited` when block content differs from expected

### Update

- [ ] `mapx agents update` replaces stale blocks in-place
- [ ] Content outside `<!-- mapx -->` block is preserved exactly
- [ ] Manually edited blocks prompt for confirmation before overwriting
- [ ] `mapx agents update --dry-run` shows diff, writes nothing

### Init integration

- [ ] `mapx init` (interactive) presents provider selection step
- [ ] Selecting providers generates the correct files
- [ ] Pressing Enter without selecting skips generation
- [ ] `mapx init --no-agents` skips provider selection entirely

### Common

- [ ] TypeScript type-check: `npx tsc --noEmit` passes with 0 errors
- [ ] All existing `mapx init`, `mapx scan`, `mapx export` behaviour unchanged

## Notes

- The `Continue` YAML format may need updates if Continue changes their config schema. Pin to Continue config spec version in the template.
- Zed's `settings.json` patch is a JSON merge — use a minimal merge that adds to `assistant.default_context` array without replacing other keys. If `.zed/settings.json` doesn't exist, create it.
- For the MCP config stanzas, include a comment noting the user should verify the binary path is in their `PATH` or use the full absolute path.
- Future: when F18 (smart context / `mapx_context` MCP tool) lands, update templates to include `mapx_context` in the MCP tool list and workflow description. This can be done via `mapx agents update`.
