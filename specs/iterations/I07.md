## Objective

Make mapx installable by any Node.js developer with a single command:

```bash
npm install -g mapx
```

And runnable with zero prior installation:

```bash
npx mapx scan
```

This is a pure developer-experience and distribution iteration. It does not change any parsing, graph, or export logic — it only changes the **build pipeline** and **package metadata** to produce a Node.js-compatible npm package and automate publishing to the npm registry.

## Scope

### Build pipeline

Add `tsup` as a devDependency. Create `tsup.config.ts` configured to transpile TypeScript to JavaScript **without bundling** (preserving directory structure), injecting a `#!/usr/bin/env node` shebang on `dist/main.js`, and emitting ESM output matching the project's `"type": "module"` setting.

The transpile-not-bundle approach is chosen deliberately: the existing WASM asset path resolution in `findAssetRoot()` relies on `dist/parsers/wasm-parser.js` being two directories below the package root. Bundling into a single file would break this math. See F13 for full analysis.

### `package.json` changes

- `bin.mapx` → `./dist/main.js` (compiled JS)
- `files` → explicit whitelist (`dist/`, `wasm/`, `queries/`, `VERSION`, `README.md`, `LICENSE`)
- `engines` → `{ "node": ">=18.0.0" }`
- `better-sqlite3` promoted from `optionalDependencies` to `dependencies` (required for Node.js runtime)
- New `build:npm` script and `prepublishOnly` hook

### CI/CD: automated npm publish

New GitHub Actions workflow (`.github/workflows/publish-npm.yml`) that:
1. Triggers on `v*.*.*` version tags
2. Installs deps, builds WASM, compiles TypeScript
3. Verifies `package.json` and `VERSION` file are in sync
4. Publishes to npm with `--provenance` for supply chain transparency

Requires `NPM_TOKEN` secret (automation token) configured in repository settings.

### `.npmignore`

Secondary guard to ensure `src/`, `specs/`, `scripts/`, `.mapx/`, and other development artifacts are never included in the published package, even if the `files` whitelist somehow misses them.

- `src/parsers/wasm-parser.ts` — comment update only, no logic changes
- All parsing, graph, export, MCP logic — zero changes

## Node.js compatibility notes

The project already supports Node.js at runtime:

| Component | Node.js status |
|-----------|----------------|
| SQLite | `NodeStore` (better-sqlite3) already implemented — runtime auto-selected |
| WASM grammars | `web-tree-sitter` is a standard npm package — Node.js compatible |
| MCP server | `@modelcontextprotocol/sdk` is Node.js native |
| CLI framework | `commander` — Node.js native |
| File I/O | Uses only `node:fs`, `node:path`, `node:readline` — ✓ |

**No runtime code changes are needed.** The Node.js path was already built into the codebase.

## Publishing checklist (one-time setup)

The following must be done once by the repository maintainer before the first automated publish:

1. Create npm account at [npmjs.com](https://www.npmjs.com)
2. Enable 2FA on the npm account
3. Create npm automation token (Profile → Access Tokens → Automation)
4. Add `NPM_TOKEN` secret to GitHub repository (Settings → Secrets → Actions)
5. Verify package name `mapx` is available: `npm info mapx`
6. If taken: update `"name"` in `package.json` to `@mapx/cli` or `mapxgraph`
7. Run `npm run build:npm && npm pack --dry-run` locally to confirm package contents
8. Publish first version manually: `npm publish --access public`
9. Verify: `npx mapx@latest --version`
10. All future publishes are automated via tag push
