## Objective

Expand the mapx MCP and CLI surface from 4 blunt tools (`mapx_scan`, `mapx_query`, `mapx_dependencies`, `mapx_export`, `mapx_status`) to a full **code intelligence API** that lets LLMs ask precise questions: "what calls this?", "what does this call?", "what breaks if I change this?", "give me everything relevant to this task".

After I11, an LLM can:
- Call `mapx_context` with a task description and receive a focused, token-efficient context block covering exactly the relevant files and symbols
- Call `mapx_callers` / `mapx_callees` to navigate call graphs without reading files
- Call `mapx_impact` to understand refactoring scope before making changes
- Call `mapx_node` to read a specific function without opening its file
- Call `mapx_files` to browse the indexed file tree without filesystem access
- See rich health information from `mapx_status` including top symbols and stale detection

## Key design decisions

### `ContextBuilder` does no NLP — only graph traversal

The context builder extracts keyword terms from the task description using simple string operations (split CamelCase/snake_case, remove stop words, lowercase) and then uses `searchSymbols()` to find matching symbols. There is no NLP library dependency. This keeps the binary small and works offline.

This means context quality is proportional to how well symbol names describe the domain. For codebases with meaningful names (most well-maintained projects), the results are excellent. For obfuscated or cryptically-named code, quality degrades gracefully — results will be less focused but still correct.

### `mapx_impact` risk model is conservative

Risk is assigned by proximity (depth) and edge type:
- `HIGH` = depth 1, data-bearing edges — always show these
- `MEDIUM` = depth 2 calls, or depth 1 import — likely to need updates
- `LOW` = depth 3+ or structural-only — awareness, not action required

The Recommendation line is:
- All HIGH risk → "Treat as BREAKING CHANGE — update all HIGH-risk callers"
- Only MEDIUM/LOW risk → "Low blast radius — proceed with caution"
- Zero affected → "No callers found — safe to change"

### `mapx_node --source` reads from disk, not the index

The `symbols` table stores `start_line` and `end_line` but not the source text. Source extraction reads the actual file at runtime. This keeps the database small and ensures the source shown is always the current version of the file (not a stale cached copy).

If the file has been modified since the last scan, `mapx_node --source` will show current source even if the index is stale. This is the desired behaviour — the source is always current; only symbol metadata (callers, callees) may be stale.

### `mapx_query` is kept as a backward-compatible alias

Existing integrations (Claude Desktop configs, scripts) that call `mapx_query` will continue to work. Internally it delegates to `mapx_search` with partial matching and no kind filter.

### Token estimation is approximate

`ContextBuilder.estimateTokens()` uses a rough heuristic:
- ~150 tokens per file (path + symbols list + edges)
- ~20 tokens per additional symbol beyond the first 3 in a file

This is intentionally conservative. Actual token counts depend on content length. The budget should be set 20% lower than the true limit to account for variability.

## Notes

- The `getTopFilesByPageRank()` and `getTopSymbolsByPageRank()` methods need PageRank scores. PageRank is currently computed in-memory on graph load (`graph.computePageRank()`). The MCP `loadContext()` function already loads the graph — these methods should call `graph.computePageRank()` once after load and expose scores via `graph.getRankedFiles()` / `graph.getRankedSymbols()`.
- `mapx_callers` and `mapx_callees` with `depth > 1` should use BFS (not DFS) for efficiency and to avoid cycles.
- The `ContextBuilder` keyword extraction should split on camelCase (`UserService` → `user`, `service`), snake_case (`user_service` → `user`, `service`), and common suffixes (`Controller`, `Service`, `Repository`, `Manager`, `Handler` → domain term extraction).
